# เรื่องเล่าของ Pi Coding Agent
## จากค้อนเล็ก ๆ สู่ Oh My Pi

* **ตัวละครหลัก**: นักพัฒนา, โมเดลภาษา, เครื่องมือชื่อ Pi, และรุ่นต่อยอดชื่อ Oh My Pi
* **คำถามใหญ่**: ทำไมบางครั้ง Agent ที่เล็กกว่า กลับทำให้เราควบคุมงานได้ดีกว่า

Note:
เปิดด้วยภาพง่าย ๆ ว่าเราไม่ได้กำลังพูดถึงหุ่นยนต์วิเศษ แต่พูดถึงเครื่องมือช่วยเขียนโค้ดที่ต้องทำงานกับมนุษย์จริง ๆ
เป้าหมายของ Slide Deck นี้คือให้คนที่ไม่เคยใช้ Coding Agent เข้าใจว่า Pi เกิดมาแก้ปัญหาอะไร และ Oh My Pi ต่อเติมอะไรจากจุดเริ่มต้นนั้น

# วันหนึ่ง คนเขียนโค้ดอยากมีผู้ช่วย
## แต่ผู้ช่วยต้องไม่แย่งพวงมาลัย

### ภาพฝัน
เราอยากพิมพ์ว่า “ช่วยแก้บั๊กนี้หน่อย” แล้วมีผู้ช่วยอ่านไฟล์ แก้โค้ด รันทดสอบ และอธิบายเหตุผลให้เข้าใจ

### ความจริง
ถ้าผู้ช่วยตัดสินใจเองมากเกินไป ซ่อนขั้นตอน หรือแก้หลายไฟล์โดยเราไม่รู้ มันไม่ใช่ผู้ช่วยแล้ว มันคือกล่องดำที่กำลังขับรถแทนเรา

Note:
เปรียบเทียบ Coding Agent กับผู้ช่วยฝึกงานที่เก่งมาก แต่ยังต้องทำงานใต้กรอบที่ชัดเจน
ผู้ช่วยที่ดีไม่ใช่คนที่ทำทุกอย่างเงียบ ๆ แต่คือคนที่ทำให้เราเห็นว่าเขาอ่านอะไร แก้อะไร และพิสูจน์ผลอย่างไร

# ปัญหาไม่ได้อยู่ที่ AI อย่างเดียว
## แต่อยู่ที่ “สายพาน” รอบตัว AI

### โมเดลภาษา
สมองที่อ่านคำสั่ง เข้าใจโค้ด และวางแผนแก้งาน

### Harness
ระบบที่ส่งไฟล์ คำสั่ง ผลลัพธ์ และเครื่องมือไปให้โมเดลใช้

### เครื่องมือ
การอ่านไฟล์ เขียนไฟล์ แก้ไฟล์ ค้นหา รันเทสต์ เปิดเบราว์เซอร์ หรือถาม IDE

Note:
คำสำคัญคือ Harness
ถ้าโมเดลคือสมอง Harness ก็คือมือ ตา หู และโต๊ะทำงานของสมองนั้น
ต่อให้สมองฉลาดมาก แต่ถ้าโต๊ะทำงานรก เครื่องมือเยอะเกินจำเป็น หรือผลลัพธ์ถูกตัดทิ้งแบบไม่บอก สมองก็ทำงานพลาดได้

# Pi เกิดจากคำถามที่เรียบง่าย
## ถ้า Agent ไม่ต้องใหญ่โตขนาดนั้นล่ะ

* **Pi**: Agent Harness และ Coding Agent CLI แบบเทอร์มินัล
* **ผู้สร้างหลัก**: Mario Zechner และชุมชนโอเพนซอร์สรอบโปรเจกต์ Pi
* **แนวคิดตั้งต้น**: ให้โมเดลใช้เครื่องมือพื้นฐานอย่างตรงไปตรงมา แล้วมนุษย์ยังเห็นทุกอย่างที่เกิดขึ้น

Note:
เล่าให้ผู้ฟังเข้าใจว่า Pi ไม่ได้เริ่มจากการแข่งทำฟีเจอร์ให้เยอะที่สุด
มันเริ่มจากความไม่พอใจกับเครื่องมือที่ซับซ้อน ซ่อนพฤติกรรม และทำให้คนเขียนโค้ดไม่รู้ว่าระบบกำลังป้อนอะไรให้ AI อยู่บ้าง

# รากฐานของ Pi คือเครื่องมือ 4 ชิ้น
## อ่าน เขียน แก้ รัน

* **read**: อ่านไฟล์หรือส่วนของไฟล์ เพื่อให้ Agent เห็นหลักฐานจริงก่อนตอบ
* **write**: สร้างหรือเขียนไฟล์ใหม่ เมื่อจำเป็นต้องวางเนื้อหาทั้งก้อน
* **edit**: ผ่าตัดแก้เฉพาะจุด ลดโอกาสทำไฟล์พังจากการเขียนทับทั้งไฟล์
* **bash**: รันคำสั่งจริง เช่น build, test, lint เพื่อพิสูจน์ว่างานใช้ได้

Note:
สำหรับมือใหม่ ให้จำง่าย ๆ ว่า Pi ให้ AI มีมือพื้นฐานสี่แบบ
มันเหมือนกล่องเครื่องมือช่างที่มีค้อน ไขควง คีม และไฟฉาย ไม่ใช่รถบรรทุกเครื่องมือทั้งคัน
ข้อดีคือ Agent ต้องคิดจากหลักฐาน และทุกการกระทำตรวจสอบได้

# หลักการที่ 3: มนุษย์ยังรับผิดชอบ
## Agent ช่วยทำงาน ไม่ใช่รับผิดแทนเรา

### Pi เลือกความเรียบง่าย
Pi ไม่มีระบบ permission หนา ๆ ในตัวแบบบังคับทุกขั้น ผู้ใช้ควรเลือก sandbox หรือ container เองเมื่อต้องการขอบเขตปลอดภัย

### บทเรียนสำหรับมือใหม่
ยิ่ง Agent เก่ง ยิ่งต้องกำหนดงานให้ชัด อ่านผลลัพธ์ และรันการตรวจสอบจริงก่อนเชื่อ

Note:
นี่ไม่ใช่การบอกให้ประมาท
แต่เป็นการแยกหน้าที่ให้ถูกต้อง เครื่องมือควรทำงานตรงไปตรงมา ส่วนเรื่องความปลอดภัยของสภาพแวดล้อมควรถูกวางด้วย Docker, sandbox, repository permissions หรือเครื่องแยกตามระดับความเสี่ยง

# Pi ทำงานเหมือนวงจรสั้น ๆ
## ขอหลักฐาน → แก้ → พิสูจน์

* **1. รับโจทย์**: ผู้ใช้บอกเป้าหมาย เช่น “แก้ปุ่ม login ที่กดแล้วค้าง”

# ความดีของ Pi อยู่ที่ความเล็ก
## เล็กพอให้เข้าใจ ใหญ่พอให้ทำงานจริง

* **ลดสิ่งลึกลับ**: เครื่องมือน้อยลง แปลว่ามีจุดที่ต้องเดาน้อยลง
* **บังคับให้ทำงานจากหลักฐาน**: อ่านไฟล์และรันคำสั่งจริง แทนการตอบจากความจำ

ในบริบทนี้เล็กแปลว่าพื้นที่หน้าตัดของระบบน้อยลง ทำให้ตรวจสอบง่ายขึ้น และลดโอกาสที่ Agent จะหลงทางเพราะเครื่องมือรอบตัวซับซ้อนเกินไป

# Pi กับ Claude Code ต่างกันตรงบุคลิก
## ไม่ใช่ใครชนะทุกสถานการณ์

### Claude Code
ผลิตภัณฑ์พร้อมใช้หลายพื้นผิว ทั้งเทอร์มินัล IDE เดสก์ท็อป เว็บ งานทีม MCP sub-agents และ automation สำหรับองค์กร

แกนเล็ก โปร่งใส ปรับแต่งได้ เหมาะกับคนที่อยากรู้ว่า Harness ทำอะไร และอยากต่อเติม workflow ของตัวเอง

### วิธีเลือก
ถ้าต้องการระบบสำเร็จรูป ใช้ง่าย และ integration เยอะ Claude Code น่าสนใจ ถ้าต้องการความโปร่งใสและควบคุมได้ลึก Pi น่าสนใจ

# Oh My Pi: Pi ที่พร้อมลุยงานจริง
## Coding Agent ที่เอา IDE, Debugger และเครื่องมือทีมมาไว้ในเทอร์มินัล

* **มันคืออะไร**: Oh My Pi คือ fork ของ Pi โดย can1357 ที่ต่อยอดจากแนวคิด “Agent ต้องโปร่งใสและควบคุมได้” แล้วเติมเครื่องมือใช้งานจริงให้ครบขึ้น
* **จุดขายหลัก**: Agent ไม่ได้แค่อ่านไฟล์แบบข้อความ แต่เชื่อมกับ IDE ผ่าน LSP จึงรู้ definition, references, diagnostics และ rename แบบเข้าใจโครงสร้างโค้ด
* **Key features**: LSP, debugger, browser automation, persistent Python/JavaScript, subagents, hash-anchored edits, web/GitHub reading และ tool harness ที่ทำงานข้ามแพลตฟอร์ม
* **เหมาะกับใคร**: คนที่ชอบความโปร่งใสของ Pi แต่ต้องการ “โต๊ะทำงานครบชุด” สำหรับโปรเจกต์จริงที่มีหลายไฟล์ หลายภาษา และต้องพิสูจน์ผลด้วยเครื่องมือจริง

Note:
เปิดบท Oh My Pi ด้วยภาพว่า นี่ไม่ใช่ Agent ตัวใหม่ที่ทิ้ง Pi แต่คือ Pi ที่ถูกต่อเติมให้พร้อมใช้งานหนักขึ้น
ถ้า Pi คือชุดเครื่องมือพื้นฐานที่สอนหลักคิด Oh My Pi คือเวิร์กสเตชันที่ต่อ IDE, debugger, browser, Python และทีมย่อยของ Agent เข้าไปใน flow เดียวกัน
ย้ำว่า key feature สำคัญที่สุดคือการเชื่อม IDE ผ่าน LSP เพราะทำให้ Agent เห็นโค้ดเป็นโครงสร้าง ไม่ใช่แค่ตัวอักษร

# Oh My Pi เติมอะไรเข้ามา
## ให้ Agent เห็นเหมือน IDE และทดสอบเหมือนวิศวกร

### LSP
รู้จักสัญลักษณ์ในโค้ดเหมือน IDE เช่น definition, references, rename และ diagnostics

### Debugger
หยุดโปรแกรม ดู stack ดูตัวแปร และหาต้นเหตุบั๊กโดยไม่ต้องเดาจาก print อย่างเดียว

### Subagents
แยกงานใหญ่เป็นงานย่อย ให้หลาย Agent สำรวจหรือแก้คนละส่วนพร้อมกัน

### Hashline Edit
แก้ไฟล์ด้วย anchor ที่ตรวจความสดของไฟล์ ลดโอกาส patch ผิดตำแหน่ง

Note:
อธิบายให้มือใหม่เห็นภาพว่าเครื่องมือเหล่านี้คือการให้ Agent มีตาแบบ IDE มีมือแบบ debugger และมีทีมย่อยช่วยทำงาน
แต่พลังที่มากขึ้นก็ต้องมาพร้อมวินัยการตรวจผลที่มากขึ้นเช่นกัน

# LSP คือสะพานเข้า IDE
## Agent ไม่ได้แค่อ่านตัวหนังสือ แต่เข้าใจโครงสร้างโค้ด

### Definition
พา Agent ไปยังจุดประกาศจริงของฟังก์ชัน คลาส type หรือ component แทนการเดาจากชื่อที่คล้ายกัน

### References
หา callsites และไฟล์ที่ใช้สัญลักษณ์เดียวกัน เพื่อรู้ว่าการเปลี่ยนหนึ่งจุดจะกระทบตรงไหนบ้าง

### Rename
เปลี่ยนชื่อแบบ symbol-aware ผ่านความรู้ของ IDE จึงปลอดภัยกว่าการค้นหาแล้วแทนข้อความธรรมดา

### Diagnostics
อ่าน error และ warning จาก language server เหมือนที่ IDE เห็น ทำให้ Agent แก้ compile/type errors จากหลักฐานจริง

Note:
LSP ย่อมาจาก Language Server Protocol ให้คิดว่าเป็นภาษากลางระหว่าง editor กับสมองวิเคราะห์โค้ดของแต่ละภาษา
ถ้า Agent มีแค่การค้นหาข้อความ มันเห็นคำว่า userId เป็นตัวอักษรเท่านั้น แต่เมื่อมี LSP มันรู้ว่า userId ตัวไหนคือ parameter ตัวไหนคือ field และตัวไหนอยู่คนละ scope
ใน Oh My Pi จุดนี้สำคัญมาก เพราะงานอย่าง rename, go to definition, find references และ diagnostics เป็นงานที่ IDE ทำได้แม่นกว่า grep ธรรมดา
ผลลัพธ์คือ Agent แก้โค้ดข้ามไฟล์ได้มั่นใจกว่า ลดโอกาสพลาด import, re-export, barrel file หรือชื่อที่ซ้ำกันคนละความหมาย

# ถ้าเพิ่งเริ่ม ควรใช้มันอย่างไร
## อย่าเริ่มจาก “ทำทั้งโปรเจกต์ให้หน่อย”

### งานเล็กก่อน
เริ่มจากบั๊กหนึ่งจุด ฟีเจอร์หนึ่งพฤติกรรม หรือไฟล์ไม่กี่ไฟล์

### ขอหลักฐาน
ให้ Agent บอกว่าอ่านไฟล์ไหน แก้อะไร และตรวจด้วยคำสั่งอะไร

### อ่านผลลัพธ์
อย่ากดผ่าน diff เพราะโค้ดที่รันใน production ยังเป็นความรับผิดชอบของมนุษย์

### เก็บบทเรียน
เมื่อ workflow ดี ให้ทำเป็น Skill หรือคำสั่งซ้ำได้ เพื่อให้ทีมใช้แนวเดียวกัน

Note:
ผู้เริ่มต้นมักพลาดเพราะมอบโจทย์กว้างเกินไป
Coding Agent เก่งที่สุดเมื่อโจทย์มีขอบเขต มีเกณฑ์ตรวจ และมีเจ้าของงานที่อ่านผลลัพธ์จริง

# เลือกเครื่องมือให้ตรงงาน
## ไม่มี Agent ตัวเดียวที่ดีที่สุดเสมอ

* **ใช้ Pi**: เมื่อต้องการเข้าใจระบบ เห็นทุกขั้น และปรับ workflow เอง
* **ใช้ Oh My Pi**: เมื่อต้องการพลังแบบ IDE, debugger, browser, subagents และเครื่องมือครบในเทอร์มินัลเดียว
* **ใช้ Claude Code**: เมื่อต้องการประสบการณ์ผลิตภัณฑ์ที่พร้อมใช้หลายช่องทางและ integration องค์กรจำนวนมาก
* **ใช้มนุษย์**: เมื่องานต้องตัดสิน product, security, architecture หรือ tradeoff ที่ไม่มีเทสต์ตัดสินแทนได้

Note:
สรุปแบบเป็นกลาง
Agent เป็นเครื่องมือ ไม่ใช่ศาสนา
เครื่องมือที่ดีคือเครื่องมือที่ทำให้ทีมส่งงานถูกต้องขึ้น เข้าใจระบบมากขึ้น และรับผิดชอบผลลัพธ์ได้มากขึ้น

# บทเรียนของ Pi
## Agent ที่ดีไม่ควรทำให้มนุษย์หายไป

* **Pi สอนว่า** ความเรียบง่ายทำให้เราเห็นกลไก และกลไกที่เห็นได้ทำให้เราเชื่อได้มากขึ้น
* **Oh My Pi สอนว่า** เมื่อเพิ่มพลัง ต้องเพิ่มความแม่นยำของเครื่องมือ ไม่ใช่เพิ่มความลึกลับ
* **Claude Code เตือนว่า** โลกผลิตภัณฑ์ต้องการความสะดวกและ integration แต่เราต้องรู้ว่า tradeoff คืออะไร
* **บทจบ**: Agent ที่ดีทำให้คนเขียนโค้ดเป็นเจ้าของงานมากขึ้น ไม่ใช่น้อยลง

Note:
ปิดด้วยประโยคจำง่าย
อย่าใช้ AI เพื่อหนีความเข้าใจ ให้ใช้ AI เพื่อเร่งเส้นทางไปสู่ความเข้าใจ
นี่คือแก่นของเรื่อง Pi และ Oh My Pi สำหรับคนเริ่มต้น
