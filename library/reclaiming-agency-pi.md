# pi: ทวงคืนอำนาจตัดสินใจของวิศวกร
## เมื่อ Coding Agent ต้องช่วยให้เราเข้าใจมากขึ้น ไม่ใช่เข้าใจน้อยลง

* เรื่องนี้ไม่ใช่ “AI ดีหรือไม่ดี”
* แต่คือ “เรายังควบคุมงานวิศวกรรมได้จริงไหม”
* pi เป็นกรณีศึกษาของ Agent Harness ที่เลือกความโปร่งใสและความเรียบง่าย

Note:
เปิดด้วยประโยคเดียว: Coding Agent ที่ดีควรทำให้วิศวกรเห็นหลักฐานมากขึ้น ไม่ใช่ซ่อนกระบวนการไว้หลังกล่องดำ
ผู้ฟังไม่จำเป็นต้องรู้จัก pi มาก่อน เพราะเราจะนิยามคำสำคัญก่อนเข้าเนื้อเรื่อง

# นิยามคำก่อนเริ่ม
## ใช้ภาษาเดียวกันก่อนเดินเรื่อง

### Coding Agent
ผู้ช่วย AI ที่อ่านโค้ด ใช้เครื่องมือ แก้ไฟล์ และรันคำสั่งเพื่อช่วยงานพัฒนา software

### Harness
โต๊ะทำงานของ Agent: ระบบที่ส่งไฟล์ เครื่องมือ คำสั่ง ผลลัพธ์ และกฎการทำงานให้โมเดลใช้

### Agency
อำนาจตัดสินใจและความเป็นเจ้าของงานของวิศวกร: รู้ว่าเกิดอะไรขึ้น ตรวจได้ และเลือกทิศทางเองได้

### Slop
งานที่ดูเหมือนเสร็จ แต่ขาดความเข้าใจ ขาดหลักฐาน ขาดการตรวจสอบ และกลายเป็นหนี้ใน codebase

Note:
คำว่า Agency ใน Deck นี้ไม่ได้แปลว่า Agent
Agency คือสิทธิ์และความสามารถของมนุษย์ในการควบคุมงานของตัวเอง
คำว่า Slop ไม่ได้หมายถึงโค้ดที่ AI เขียนทั้งหมด แต่หมายถึงผลลัพธ์ที่ไม่มีเจ้าของและไม่มีการตรวจจริง

# เราได้ผู้ช่วยที่เร็วขึ้น
## แต่บางครั้งกลับตรวจงานยากขึ้น

### สิ่งที่เราต้องการ
สั่งงานเป็นภาษาไทยหรืออังกฤษ แล้ว Agent อ่านไฟล์ แก้โค้ด รันทดสอบ และสรุปเหตุผล

### สิ่งที่ทำให้เสี่ยง
ถ้าเราไม่เห็นว่า Agent อ่านอะไร แก้ตรงไหน หรือพิสูจน์อย่างไร ความเร็วจะเปลี่ยนเป็นความไม่แน่นอน

Note:
เล่าเป็นเรื่องของวิศวกรคนหนึ่งที่อยากให้ Agent แก้บั๊กเล็ก ๆ
ตอนแรกเหมือนประหยัดเวลา แต่เมื่อ diff ใหญ่ขึ้นและขั้นตอนถูกซ่อน เขาเริ่มไม่รู้ว่างานถูกจริงหรือแค่ดูถูก

# ปัญหาไม่ได้อยู่ที่โมเดลอย่างเดียว
## Harness เป็นตัวกำหนดว่า Agent เห็นอะไร และพิสูจน์อะไรได้

* โมเดลคือสมอง
* Harness คือมือ ตา หู และโต๊ะทำงาน
* เครื่องมือที่ซ่อนข้อมูล ทำให้คำตอบตรวจยาก
* เครื่องมือที่โปร่งใส ทำให้มนุษย์ตรวจซ้ำได้

Note:
ถ้า Agent ตอบผิด เรามักโทษโมเดลทันที
แต่หลายครั้งปัญหาเกิดจาก Harness: ตัด output, ป้อน context ไม่ครบ, แก้ไฟล์แบบไม่ anchored, หรือไม่บังคับให้รันทดสอบ
นี่คือเหตุผลที่ pi น่าสนใจ เพราะมันถามกลับว่า Harness ควรเล็กและชัดแค่ไหน

# pi เลือกคำตอบที่ย้อนกระแส
## เพิ่มความน่าเชื่อถือด้วยการลดสิ่งที่ซ่อนอยู่

### เครื่องมือใหญ่ทั่วไป
ให้ความสะดวกมาก แต่บางครั้งซ่อน context, ซ่อน orchestration, และซ่อน tradeoff

### แนวคิดของ pi
ให้ Agent ใช้เครื่องมือพื้นฐานอย่างตรงไปตรงมา และให้มนุษย์เห็นหลักฐานที่เกิดขึ้นระหว่างทาง

Note:
นี่คือจังหวะ storytelling สำคัญ
ไม่ต้องบอกว่าเครื่องมือใหญ่ผิดเสมอ ให้บอกว่ามันมี tradeoff
pi เลือก tradeoff อีกฝั่ง: ยอมให้ระบบดูเรียบกว่า เพื่อให้ตรวจง่ายกว่า

# เครื่องมือหลักของ Agent ควรตอบ 4 คำถาม
## เห็นอะไร แก้อะไร เขียนอะไร พิสูจน์อะไร

* **read**: อ่านไฟล์หรือช่วงบรรทัด เพื่อเริ่มจากหลักฐานจริง
* **edit**: แก้เฉพาะจุด ลดความเสี่ยงจากการเขียนทับทั้งไฟล์
* **write**: สร้างหรือเขียนไฟล์ใหม่ เมื่อจำเป็นต้องวางเนื้อหาทั้งก้อน
* **bash**: รัน build, test, lint หรือคำสั่งจริงเพื่อพิสูจน์ผล

Note:
อย่าเล่าเครื่องมือเป็น API แห้ง ๆ
ให้เล่าว่า read คือการดูหลักฐาน edit คือการผ่าตัด write คือการวางชิ้นงานใหม่ และ bash คือการทดลองจริง
ผู้ฟังจะจำได้ว่า Agent ที่ดีต้องปิดงานด้วย evidence ไม่ใช่คำรับรอง

# Agency กลับมาเมื่อเราตรวจซ้ำได้
## ความเร็วต้องไม่แลกกับการมองไม่เห็น

### เห็นบริบท
รู้ว่า Agent อ่านไฟล์ไหนและใช้ข้อมูลอะไรในการตัดสินใจ

### เห็นการเปลี่ยนแปลง
diff ต้องเล็กพอให้มนุษย์อ่านและโยงกับโจทย์ได้

### เห็นการพิสูจน์
มี command หรือ test ที่ยืนยันพฤติกรรม ไม่ใช่แค่คำว่า “น่าจะผ่าน”

### เห็นข้อจำกัด
Agent ต้องบอกสิ่งที่ยังไม่ได้ตรวจ ไม่ใช่พูดเหมือนทุกอย่างแน่นอน

Note:
นี่คือหัวใจของ Deck
Agency ไม่ได้แปลว่าทำเองทุกบรรทัด แต่แปลว่าเราไม่สูญเสียความสามารถในการตรวจและตัดสินใจ

# โปร่งใสไม่ได้แปลว่าปลอดภัยโดยอัตโนมัติ
## ขอบเขตความเสี่ยงต้องออกแบบแยกต่างหาก

### ข้อมูลที่ตรวจแล้ว
README ของ pi ระบุว่า pi ไม่มี built-in permission system สำหรับจำกัด filesystem, process, network หรือ credential access โดย default

### บทเรียน
ถ้างานเสี่ยง ให้ใช้ container, sandbox, repository permissions หรือเครื่องแยก ไม่ใช่หวังว่า Agent จะไม่พลาด

Note:
สไลด์นี้ทำให้เรื่องไม่สวยเกินจริง
pi เลือกความเรียบง่าย แต่ความเรียบง่ายต้องมาพร้อมวินัยของคนใช้
ผู้ฟังควรจำว่า safety boundary เป็นความรับผิดชอบของ environment

# แล้วเครื่องมือใหญ่ผิดไหม?
## ไม่ผิด แต่โจทย์คนละแบบ

### Claude Code

# LSP คืออะไร ทำไมเกี่ยวกับ Agency
## Agent เริ่มเห็นโค้ดแบบ IDE ไม่ใช่แค่ข้อความ

### Definition
ไปยังจุดประกาศจริงของ function, class, type หรือ component

### References
เห็น callsites ก่อนแก้ symbol ที่ใช้หลายไฟล์

### Rename
เปลี่ยนชื่อแบบเข้าใจ scope ดีกว่าค้นหาแล้วแทนคำธรรมดา

### Diagnostics
อ่าน error และ warning เหมือนที่ IDE เห็น

Note:
ยกตัวอย่างคำว่า userId
ถ้า search แบบข้อความจะเห็นทุกคำเหมือนกัน แต่ LSP ช่วยแยกบริบทว่าเป็น parameter, field, import หรือ symbol คนละตัว
นี่ทำให้ Agent แก้โค้ดแบบมีหลักฐานมากขึ้น

# วิธีใช้ Agent โดยไม่เสีย Agency
## ให้โจทย์เล็ก มีขอบเขต และตรวจได้

### Prompt ที่เสี่ยง
“ทำทั้งโปรเจกต์ให้ดีขึ้น” — กว้างเกินไป วัดผลไม่ได้ และเปิดทางให้ diff บวม

### Prompt ที่ดีขึ้น
“แก้ bug ปุ่ม login ค้างเมื่อ API ตอบ 401 อ่านไฟล์ที่เกี่ยวข้อง แก้เฉพาะ flow นี้ และรันทดสอบที่ครอบคลุม branch นี้”

Note:
ให้ผู้ฟังเห็นว่าคุณภาพผลลัพธ์ไม่ได้มาจาก tool อย่างเดียว แต่มาจากโจทย์ที่วัดผลได้ด้วย
Agent เก่งที่สุดเมื่อมีเป้าหมาย ขอบเขต และเกณฑ์ตรวจ

# บทเรียนของ pi
## อย่าใช้ AI เพื่อหนีความเข้าใจ ให้ใช้เพื่อไปถึงความเข้าใจเร็วขึ้น

* ความเร็วมีค่าก็ต่อเมื่อผลลัพธ์ตรวจได้
* เครื่องมือที่ดีทำให้มนุษย์เห็นหลักฐานมากขึ้น

Note:
ปิดด้วยภาพชัด ๆ ว่า Agent ไม่ใช่คนรับผิดแทนเรา
มันควรเป็นเครื่องมือขยายกำลังของวิศวกร ไม่ใช่เครื่องมือทำให้เราเลิกเข้าใจระบบของตัวเอง
